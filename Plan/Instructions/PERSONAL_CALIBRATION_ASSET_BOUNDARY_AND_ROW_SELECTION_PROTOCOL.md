# Personal Calibration Asset Boundary And Row Selection Protocol

Updated: 2026-07-16 America/Chicago

## Purpose

Keep reusable project development separate from operator-owned character workflows, references, prompt profiles, generated candidates, models, and local calibration packages.

## Classification

Character1-specific assets under local `ztest`, Character1 workflow/profile paths, and their derived comparison media are classified as `personal_calibration_noncanonical` unless a later explicit project intake promotes a generalized, portable fixture.

These assets may be used to:

- test reusable workflow mechanics;
- expose identity, anatomy, regional-edit, or preservation defects;
- develop deterministic crop, comparison, and validation tooling;
- detect and prevent duplicate generation.

They must not:

- update a generic project Item or Tracker row by themselves;
- become identity, mask, geometry, production, or certification authority;
- redefine a generic project identifier such as `C01` as the operator's personal Character1;
- be treated as reproducible from a clean clone while any required input remains local or untracked;
- be staged, moved, deleted, or modified as part of generic project work without explicit user direction.

Derived comparison crops and sheets may be preserved as local negative calibration evidence. Their classification remains `personal_calibration_noncanonical`, and they do not change generic Row010 authority.

## Next-Row Selection Gate

Before selecting implementation work, reject a candidate row when any condition below is true:

1. The row is completed or its exact proof is marked immutable or no-rerun.
2. The only remaining action requires human-only playback or production authority and no new machine-verifiable input exists.
3. The row depends on unavailable trusted gold masks or another explicitly blocked authority.
4. The row is owned by another active task or side session.
5. Generic proof would depend on personal, local-only, or untracked assets.
6. The proposed action only refreshes hydration, manifests, evidence indexes, or ledgers without a new runtime artifact, executable capability, measured quality change, or exact productive blocker.

Select the highest-priority remaining row only after this gate passes. Batch three to five compatible implementation and QA units when practical. Hydration and tracking updates follow the implementation outcome and are not the work unit.
